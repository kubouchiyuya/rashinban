# 実行時の動き

goal-setter は、Goal の文章を作るスキルです。新しい実行の仕組みを追加するものではありません。Codex や Claude Code に元からある機能を使います。

## 発動のしかた

| 場面 | 動き |
| --- | --- |
| Codex で、別の確認役や別スレッドが不要 | Codex の Goal 機能でそのまま設定できる |
| Codex で、`spawn_agent` または `create_thread` が必要 | ユーザーが送る `/goal ...` 行を返す |
| Claude Code | ユーザーが送る `/goal ...` 行を返す |

Codex では、`spawn_agent` と `create_thread` はユーザー自身の明示した依頼から使われます。そのため goal-setter は `/goal ...` の一行を渡し、ユーザーがそれを送る形にします。これは、`spawn_agent` を最後の確認だけに使う場合も同じです。

## 分けて進める作業

作業を分けるのは、次のような場合だけです。

- それぞれの成果が独立している
- それぞれを別々に確認できる
- 分ける手間より、短縮できる時間の方が大きい
- 最後にまとめる方法が見えている

分けられるかどうかは、ファイルの場所だけでは決めません。実際の動きがどれだけつながっているか、同じ状態を触るか、あとでまとめる時に壊れやすいかを先に見ます。ユーザーが数を指定していない限り、別の確認役の数は固定しません。親になる作業が、必要な数だけ呼び、結果をまとめてから次を判断します。

## 長い作業

長い自律作業では、Goal に「未確認一覧を更新しながら進む」動きを入れられます。実行する側は `execution-notes.md` に、残っている確認、見た証拠、合格・不合格・保留・詰まり、途中の判断を短く残します。確認するたびに状態を更新し、完了条件を満たしておらず、止まる条件にも当たらないなら、次に一番危ない未確認項目か、一番あいまいな未確認項目へ進みます。「次にやること」を書いただけでは止まりません。

## 深掘り確認

goal-setter が質問するのは、答えによって次のどれかが変わる時だけです。

- 完了条件
- 確認方法
- 作業範囲
- 危険度
- 止まる条件

ユーザーが「詰めて確認して」と言った場合も、重要な質問を一つずつ出します。おすすめ回答も添えます。コード、ドキュメント、資料を読めば分かることは、質問する前に確認します。

## Codex

`create_thread` は、次がすべて成り立つ時だけ使います。

- 独立した書き込み作業が2つ以上ある
- それぞれの担当範囲と確認方法がはっきりしている
- 共有部分をどうつなぐかが分かっている
- 別スレッド化で増える確認・統合作業より、短縮できる時間の方が大きい
- 既に Git の作業場所を分けられる状態になっている

並列化のためだけに Git を初期化したり、土台を作ったり、共有部分を設計したりしません。作業場所がまだ向いていない場合は、直列で進めるか、構造を変える前に確認します。

`spawn_agent` は、別の読み取り確認で完了判断が変わり得る時に使います。たとえば調査、複数の観点からの確認、結論を崩しにいく確認、最後の確認、ログの読み取り、既存の動きの把握などです。低リスクで十分な自動確認がある作業では不要です。別の確認役は証拠、反証、不明点、抜けを返し、親になる作業がまとめ、修正判断、最終判断を持ちます。

## Claude Code

Claude Code でも同じ考え方で動かせます。Goal は、何を分けて確認するかを書きます。細かい実行方法は Claude Code 側に任せます。

## Goal が扱うこと

大きめの作業では、goal-setter は次を考えます。

- 何を達成したいか
- 何ができたら完了か
- 何で確認するか
- どこまで触ってよいか
- 何を変えてはいけないか
- 既に使われている表、資料、管理ファイルを増やさず更新するか
- どんな時に質問するか
- どこから読むか
- 必要な確認を弱めずに済ませるルール
- 長い作業で未確認一覧を更新しながら進むこと
- 止まる条件
- 完了前にどの強さで見直すか

短い作業には短い Goal を書きます。結果に影響しない内容は入れません。
